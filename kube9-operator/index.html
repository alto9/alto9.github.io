Overview

Kube9 Operator is a Kubernetes operator that extends and enhances the capabilities of the Kube9 VS Code Extension. It runs inside your Kubernetes cluster and provides advanced automation, monitoring, and management features that complement the client-side extension.

By deploying the Kube9 Operator in your cluster, you unlock:

  • Event Tracking & History - Comprehensive event system with CLI query interface (free tier, available now)
  • Cluster Event Recording - Automatically captures significant Kubernetes events (Pod failures, node issues, etc.)
  • Operator Insights - Records operator activities, assessments, and health transitions
  • Dual-tier Security - Free tier CVE scanning + Pro tier configuration analysis (planned)
  • Automated Optimization - Resource optimization and recommendations (planned)
  • Team Collaboration - Shared insights and configurations (planned)

Features

Event Tracking

Available

Comprehensive event recording system that captures Kubernetes events (Pod failures, node issues, OOMKills) and operator activities in a local SQLite database.

CLI Query Interface

Available

Powerful CLI for querying events with filtering by type, severity, date range, and Kubernetes objects. Supports JSON, YAML, and table output formats.

Event Retention

Available

Automatic cleanup with configurable retention policies: 7 days for info/warning events, 30 days for error/critical events. Keeps your database size manageable.

Security Scanning

Planned

Comprehensive security through dual-tier architecture: Free tier CVE vulnerability scanning (Trivy/Grype) plus Pro tier configuration security analysis for misconfigurations, exposure risks, and RBAC issues.

Automated Optimization

Planned

Continuously monitor cluster resources and automatically apply AI-recommended optimizations for performance and cost efficiency.

Team Collaboration

Planned

Share AI insights, recommendations, and configurations across your team through custom resources.

Custom Resource Definitions

The operator introduces several CRDs to manage kube9-specific workflows:

  • kube9Analysis: Stores AI analysis results for cluster resources
  • kube9Optimization: Defines optimization recommendations and tracks application
  • kube9SecurityScan: Manages CVE vulnerability scanning schedules and results (Free tier with Trivy/Grype)
  • kube9Policy: Defines organizational policies and compliance rules
  • kube9Insight: Stores team-shared insights including AI security configuration analysis

Getting Started

Prerequisites

Kube9 Operator requires a running Kubernetes cluster (1.20+) and kubectl configured for cluster access.

Installation

Once released, you'll be able to install Kube9 Operator using Helm:

# Add the Kube9 Helm repository
# Note: charts.kube9.io aliases app.kube9.io (no authentication required)
helm repo add kube9 https://charts.kube9.io
helm repo update
# Install the operator
helm install kube9-operator kube9/kube9-operator \
  --namespace kube9-system \
  --create-namespace
# Verify installation
kubectl get pods -n kube9-system

Configuration

Configure the operator through the Helm values file:

# values.yaml
replicaCount: 1
operator:
# Enable AI analysis features
aiAnalysis:
enabled: true
interval: 5m
# Security scanning configuration
securityScanning:
enabled: true
schedule: "0 */6 * * *"
# Optimization features
optimization:
enabled: true
autoApply: false  # Require manual approval
# Connection to Kube9 Server
kube9Server:
url: "https://api.kube9.io"
# API key stored as secret

Integration with VS Code Extension

The VS Code extension automatically detects when the operator is installed in your cluster and enables additional features:

  1. Open Kube9 in VS Code
  2. Connect to a cluster with the operator installed
  3. Look for the "Operator Enabled" badge in the cluster view
  4. Access operator features through the context menu

Event System & CLI Query Interface

The operator includes a comprehensive event tracking system that records cluster events, operator activities, and insights in a local SQLite database. Query events using the built-in CLI interface.

Event Storage

  • Database: SQLite with WAL mode for concurrent access
  • Location: /data/kube9.db (PersistentVolume)
  • Retention: 7 days for info/warning, 30 days for error/critical
  • Event Types: Cluster events (Pod failures, Node issues), operator insights, assessments, health transitions

CLI Query Interface

Query events directly from the operator pod using kubectl exec:

# Get operator status
kubectl exec -n kube9-system deploy/kube9-operator -- \
  kube9-operator query status
# List all events (JSON output)
kubectl exec -n kube9-system deploy/kube9-operator -- \
  kube9-operator query events list
# Filter events by severity and type
kubectl exec -n kube9-system deploy/kube9-operator -- \
  kube9-operator query events list \
--severity error \
--type cluster
# Filter by date range
kubectl exec -n kube9-system deploy/kube9-operator -- \
  kube9-operator query events list \
--since "2025-12-01T00:00:00Z" \
--until "2025-12-02T00:00:00Z"
# Filter by Kubernetes object
kubectl exec -n kube9-system deploy/kube9-operator -- \
  kube9-operator query events list \
--object-kind Pod \
--object-namespace default \
--object-name my-pod
# Get single event by ID
kubectl exec -n kube9-system deploy/kube9-operator -- \
  kube9-operator query events get evt_20251202_103045_a7f3b9
# Output in table format (human-readable)
kubectl exec -n kube9-system deploy/kube9-operator -- \
  kube9-operator query events list --format table
# Output in YAML format
kubectl exec -n kube9-system deploy/kube9-operator -- \
  kube9-operator query events list --format yaml
# Pagination
kubectl exec -n kube9-system deploy/kube9-operator -- \
  kube9-operator query events list --limit 100 --offset 50

Query Options

Option Description Values
--type Filter by event type cluster, operator, insight, assessment, health, system
--severity Filter by severity level info, warning, error, critical
--since Events since date (ISO 8601) e.g., 2025-12-01T00:00:00Z
--until Events until date (ISO 8601) e.g., 2025-12-02T23:59:59Z
--object-kind Filter by Kubernetes resource kind Pod, Node, Deployment, etc.
--object-namespace Filter by namespace Any namespace name
--object-name Filter by resource name Any resource name
--limit Maximum results to return 1-1000 (default: 50)
--offset Number of results to skip Any non-negative integer
--format Output format json (default), yaml, table

RBAC Requirements

To query events via kubectl exec, you need the following permissions:

apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
name: kube9-query-access
namespace: kube9-system
rules:
- apiGroups: [""]
resources: ["pods/exec"]
verbs: ["create"]

VS Code Extension Integration

The Kube9 VS Code Extension will automatically use these CLI commands to provide an Event Viewer interface, allowing you to browse cluster events directly from VS Code without needing to run kubectl commands manually.

Free Tier Feature

Event tracking and CLI queries are available in the free tier and do not require an API key. All data stays local in your cluster.
