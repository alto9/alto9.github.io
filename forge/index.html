Overview

Forge Studio is a VS Code and Cursor extension that helps teams keep product vision, technical contracts, and roadmaps in sync—then drive GitHub issues through a staged agent workflow from refinement to implementation and review.

Running Forge: Initialize Agents installs shared Cursor/VS Code assets under ~/.cursor/ and project context under .forge/, including a skill registry so agents run the right scripts for tests, lint, branches, and GitHub operations.

  • Agentic SDLC — Product Owner, Architect, Planner, Technical Writer, domain experts, Engineer, and Quality Assurance roles encoded as agents
  • GitHub-native flow — Commands for roadmap sync, issue refinement, implementation from an issue link, and PR review
  • Quality gates — Engineer runs project test and lint skills from .forge/skill_registry.json and does not commit until checks pass
  • Branch discipline — Technical Writer creates and links the parent issue branch; implementation branches for parent or sub-issues are created during build

What is context engineering here?

Forge treats context as code: vision.json, a knowledge_map.json, and domain docs under .forge/ give agents a single place to read constraints. That context flows into GitHub issues and into implementation, so AI-assisted work stays aligned with what the team agreed.

Key Features

Initialize Agents

Detects Cursor vs VS Code, copies agents, commands, skills, and hooks into ~/.cursor/, and seeds .forge/ in the repo.

Staged agent workflow

From vision and architecture through planning, ticket refinement, implementation, and QA—each phase has a dedicated agent and handoff.

Slash commands

/architect-this, /plan-roadmap, /refine-issue, /build-from-github, /review-pr orchestrate the flow with your repo.

Chat participants

In VS Code chat, use @forge, @forge-refine, @forge-build-issue, @forge-review-pr, and helpers for commit, push, and PR creation.

Skill registry

Maps agents to scripts for create-feature-branch, push-branch, unit-test, integration-test, lint-test, and more—so validation is explicit and repeatable.

Domain SMEs

Runtime, business logic, data, interface, integration, and operations subagents maintain contracts under .forge/ when the Architect delegates.

End-to-end workflow

Typical flow from idea to merged PR (details live in the repo’s AGENT_FLOW.md):

1. Product Owner

Align vision.json and project config with strategy and market input.

2. Architect (+ domain SMEs)

Update the knowledge map and domain contracts; invoke SMEs (runtime, data, interface, etc.) when scope touches their areas.

3. Planner

Sync GitHub milestones and issues from the roadmap using milestone and issue skills.

4. Technical Writer (/refine-issue)

Refine the GitHub issue with SMEs, create the parent branch from main, push it, and link it to the issue. Optional sub-issues stay on GitHub only—no separate git branch per sub-issue.

5. Engineer (/build-from-github)

Create or checkout feature/issue-<N> (from main or the parent branch for sub-issues), link it to the issue, implement, then run all configured unit, integration, and lint skills until green before commit and PR.

6. Quality Assurance (/review-pr)

Review the PR for correctness and security; post feedback. A human merges when satisfied.

Getting Started

Installation

From source

git clone https://github.com/alto9/forge.git
cd forge
npm install
npm run build
npm run package
code --install-extension forge-studio-*.vsix

Requires Node.js 22.14+ and npm 10+ per the extension manifest. See the repository README if paths differ by version.

First steps in a project

  1. Open your repository in VS Code or Cursor.
  2. Command Palette (Cmd/Ctrl + Shift + P) → Forge: Initialize Agents.
  3. Confirm creation of ~/.cursor/ assets and .forge/ in the project.
  4. Edit .forge/vision.json and domain docs; use slash commands or chat participants to run the workflow.

Architecture

Where files live

After Initialize Agents:

~/.cursor/
agents/          # product-owner, architect, planner, tech-writer, engineer, quality-assurance, SMEs
commands/        # architect-this, plan-roadmap, refine-issue, build-from-github, review-pr
skills/          # create-feature-branch, push-branch, tests, lint, …
hooks/           # e.g. JSON schema validation for .forge
hooks.json
your-project/.forge/
vision.json           # Product vision
project.json          # GitHub URLs, project metadata
skill_registry.json   # Agent → skill assignments (copied from Forge templates)
knowledge_map.json    # Domain layout
schemas/              # JSON Schema for .forge files
runtime/, business_logic/, data/, …  # Domain contract docs

Extension

  • Registers chat participants and contributes setup commands
  • Copies workflow templates from the packaged resources/workflow tree into the user and project locations above

Best practices

  1. Keep vision and knowledge map current before large roadmap or refine passes.
  2. Let Technical Writer own the parent branch; let Engineer create implementation branches per issue.
  3. Trust the skill registry for test and lint commands instead of ad-hoc scripts.
  4. Do not skip green checks—the intended flow blocks commit/PR until tests and lint pass.
  5. Re-run Initialize Agents after upgrading Forge so skills and agents stay in sync.

Technology stack

  • Platform: VS Code Extension API (Cursor-compatible)
  • Language: TypeScript
  • Runtime: Node.js 22.14+ (see repo engines)
  • Build: webpack / project scripts as defined in the repository
